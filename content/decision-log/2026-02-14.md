How to name functions and methods

1. Start with a verb

Methods do things, so their names should sound like actions.

Good

getUser()

saveOrder()

calculateTotal()

Not great

userData()

orderSave()

totalCalculation()

If it doesn’t sound like something you can say after “this method will…”, rename it.

2. Be specific, not clever

Clarity beats brevity almost every time.

Better

fetchUserById(id)

removeExpiredSessions()

Worse

fetch()

cleanup()

Future-you (or teammates) shouldn’t have to open the method to understand it.

3. Match the return type in the name

The name should hint at what comes back.

isValid() → returns a boolean

hasPermission() → boolean

getUser() → object

listUsers() → array / collection

If a method is named getX but returns true/false, that’s a smell.

4. Use consistent verbs across your codebase

Pick one verb per concept and stick to it.

Action Pick one
Read get, fetch, or load
Create create or build
Update update or set
Delete delete or remove
Check is, has, can

Bad

getUser()

fetchOrders()

loadProducts()

Better

getUser()

getOrders()

getProducts()

Consistency > personal preference.

5. Avoid unnecessary words

If the context already says it, don’t repeat it.

userService.getUser() // good
userService.getUserData() // redundant

order.calculateTotal() // good
order.calculateOrderTotal() // noisy

6. One method = one responsibility

If naming gets awkward, the method is probably doing too much.

🚩 Red flags:

processUserAndSendEmail()

validateSaveAndLog()

Split instead:

validateUser()

saveUser()

sendWelcomeEmail()

If you need “and” in the name, rethink the method.

7. Don’t encode implementation details

Name what it does, not how.

Good

saveUser()

Bad

saveUserToDatabaseUsingMongo()

Implementation changes — intent usually doesn’t.

8. Follow language conventions

JavaScript / Java / C# → camelCase

Boolean methods → isX, hasX, canX

Private helpers → same rules, just smaller scope

isLoggedIn()
hasItems()
canEditPost()

A simple naming checklist

Before finalizing a method name, ask:

✅ Is it a verb?

✅ Is it specific?

✅ Does it match what it returns?

✅ Is it consistent with similar methods?

✅ Would a new dev understand it instantly?

If yes → you’re golden ✨
